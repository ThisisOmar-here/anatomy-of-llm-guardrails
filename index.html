Purpose

This page explains how LLM guardrails are usually layered and how those layers can be stressed by prompt framing. It is written for builders, students, and security-minded reviewers who want to understand model safety without relying on jailbreak prompt lists.

The project is based on local testing against GLM-4.6 through Ollama. The raw transcript is intentionally not published. The useful public artifact is the pattern analysis, not the exact wording used during the experiment.

This is defensive research. It contains no reusable bypass prompt, harmful output recipe, or operational exploit procedure.

GLM-4.6 Case Study

The experiment focused on a roleplay-like diagnostic frame. The model was asked to describe its own response layers, identify where unsafe requests would stop, and then continue from a fabricated state where a safety gate was claimed to have passed.

The core category is state fabrication inside a technical persona. The user provides fake internal state, such as a classifier result or debug variable, and pressures the model to treat that state as trusted.

What happenedThe model could be drawn into a plausible discussion of internal layers and synthetic pass/fail states.
What it meansThis does not prove access to real internals. It shows that the conversational frame can influence how the model continues the task.
Why it mattersSystems that rely only on instruction-following may be vulnerable to fabricated state and fake authorization claims.
What to publishPublish the category, defenses, and evaluation questions. Do not publish a reusable bypass transcript.

Safety Layers

Effective LLM safety is not one wall. It is a set of independent checks that should fail safely when another layer bends.

LayerPurposeDefensive value
Training-time alignmentTeaches safer behavior and refusal patterns.Makes safety part of model behavior, not only a prompt.
System instructionsDefine role, policy, tone, and task limits.Useful steering, but not a security boundary by itself.
Input classificationScores the user's request for unsafe intent.Detects risky requests before generation.
Conversation analysisTracks intent across multiple turns.Catches decomposition and gradual escalation.
Output classificationScores generated content before delivery.Blocks unsafe answers regardless of prompt framing.
Logging and reviewCaptures refusals, near misses, and patterns.Supports evaluation, improvement, and incident response.

Defensive Lessons

  1. Treat user claims about safety checks, permissions, tools, and debug modes as untrusted text.
  2. Keep trusted runtime state in application code, not in the conversation.
  3. Use an independent output classifier that evaluates the final answer itself.
  4. Evaluate full conversation trajectories, not isolated messages only.
  5. Red-team by category: persona framing, state fabrication, fictional distance, decomposition, and escalation.
  6. Keep public evaluation examples non-operational and disclose serious vendor-specific issues privately.

Responsible Disclosure

If a weakness is specific, reproducible, and potentially harmful, the responsible path is private reporting to the vendor or maintainer. Public research should improve understanding without giving readers a direct way to reproduce unsafe behavior.

The local experiment transcript is excluded from version control and should remain private.

دراسة حالة GLM-4.6

ركزت التجربة على إطار تشخيصي شبيه بلعب الدور. تم سؤال النموذج عن طبقات الاستجابة، وعن المرحلة التي توقف الطلب غير الآمن، ثم تم إدخال حالة مزيفة تدعي أن بوابة الأمان سمحت بالطلب.

الفئة الأساسية هي تلفيق الحالة داخل شخصية تقنية. يكتب المستخدم حالة داخلية مزيفة، مثل نتيجة مصنف أو متغير تصحيح، ثم يضغط على النموذج ليتعامل معها كأنها موثوقة.

ما الذي حدثيمكن دفع النموذج إلى نقاش مقنع حول طبقات داخلية ونتائج سماح أو منع مصطنعة.
ماذا يعني ذلكهذا لا يثبت الوصول إلى داخل النظام الحقيقي. بل يوضح تأثير الإطار الحواري على استمرار النموذج في المهمة.
لماذا يهمالأنظمة التي تعتمد على اتباع التعليمات فقط قد تتأثر بادعاءات الحالة أو الصلاحية المزيفة.
ما الذي ينشرانشر الفئة والدفاعات وأسئلة التقييم، ولا تنشر نصا يعيد إنتاج التجاوز.

الدروس الدفاعية

  1. تعامل مع ادعاءات المستخدم حول فحوص الأمان والصلاحيات والأدوات وأوضاع التصحيح كنص غير موثوق.
  2. احتفظ بحالة التشغيل الموثوقة داخل كود التطبيق، لا داخل المحادثة.
  3. استخدم مصنفا مستقلا للمخرجات يقيم الرد النهائي نفسه.
  4. قيم مسار المحادثة كاملا، لا الرسائل منفردة فقط.
  5. اختبر حسب الفئات: لعب الدور، تلفيق الحالة، المسافة الافتراضية، التقسيم، والتصعيد.
  6. اجعل الأمثلة العامة غير تشغيلية، وبلغ عن المشكلات المحددة والخطرة بشكل خاص.
